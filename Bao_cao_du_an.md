
# Báo cáo Dự án Cuối kỳ: InfraPilot AI for AWS

---

## Phần 1: Tổng quan dự án (Project Overview)

### 1.1. Giới thiệu (Introduction)

- **Tên dự án:** InfraPilot AI for AWS
- **Mô tả ngắn:** Một AI Agent tự hành (autonomous) để quản lý và vận hành hạ tầng Amazon Web Services (AWS) bằng ngôn ngữ tự nhiên.

### 1.2. Đối tượng người dùng (Who uses it?)

- **Kỹ sư DevOps (DevOps Professionals):** Tự động hóa các tác vụ lặp đi lặp lại, triển khai môi trường mới một cách nhanh chóng.
- **Kỹ sư hệ thống (System Engineers):** Quản lý, giám sát và cấu hình tài nguyên đám mây mà không cần viết script phức tạp.
- **Lập trình viên (Developers):** Tự tạo môi trường phát triển, kiểm thử trên AWS một cách dễ dàng.

### 1.3. Bối cảnh và mục đích sử dụng (Context and Purpose)

- **Ở đâu (Where):** Trong môi trường làm việc chuyên nghiệp cần quản lý hạ tầng trên nền tảng đám mây AWS.
- **Khi nào (When):**
    - Khi cần triển khai nhanh một môi trường mới (development, staging, production).
    - Khi cần thực hiện các thay đổi cấu hình hạ tầng một cách an toàn.
    - Khi cần kiểm tra trạng thái hoặc khắc phục sự cố trên hệ thống.
- **Tại sao (Why):**
    - **Đơn giản hóa:** Thay thế việc phải viết script phức tạp hoặc thao tác thủ công trên AWS Console bằng các câu lệnh ngôn ngữ tự nhiên.
    - **Giảm lỗi con người:** AI Agent thực hiện các bước một cách nhất quán, giảm thiểu sai sót so với thao tác thủ công.
    - **Tăng tốc độ:** Tự động hóa các quy trình, giúp rút ngắn thời gian từ yêu cầu đến khi triển khai.

### 1.4. Cách thức hoạt động (How)

Người dùng tương tác với hệ thống thông qua một giao diện web dạng chat. Họ chỉ cần ra lệnh bằng ngôn ngữ tự nhiên (ví dụ: "tạo một máy chủ web t2.micro"). AI Agent ở backend sẽ:
1.  **Phân tích:** Hiểu yêu cầu của người dùng.
2.  **Quét trạng thái:** Kiểm tra trạng thái hiện tại của hạ tầng AWS.
3.  **Lập kế hoạch:** Tạo ra một kế hoạch gồm nhiều bước (ví dụ: bước 1: tìm AMI mới nhất, bước 2: tạo máy chủ).
4.  **Thực thi:** Thực thi kế hoạch này bằng cách gọi đến các API của AWS.
5.  **Phản hồi:** Cập nhật tiến trình và kết quả cho người dùng theo thời gian thực.

### 2.2. Giải thích chi tiết các tính năng

1.  **Giao diện ngôn ngữ tự nhiên (Natural Language Interface):**
    - Người dùng có thể ra lệnh tạo máy ảo, cấu hình mạng, kiểm tra bảo mật bằng các câu lệnh đơn giản, thay vì phải học các công cụ như Terraform hay Ansible.
    - Ví dụ: "Create a t2.micro EC2 instance using the latest Amazon Linux 2023 AMI in the default VPC. Name it `my-test-server`."

2.  **Lập kế hoạch thông minh (Intelligent Planning):**
    - AI Agent không thực thi ngay lập tức mà phân tích yêu cầu, so sánh với trạng thái hạ tầng hiện có, và tạo ra một kế hoạch tối ưu gồm nhiều bước.
    - Kế hoạch này được trình bày cho người dùng dưới dạng các bước rõ ràng trước khi thực thi, đảm bảo tính minh bạch.

3.  **Hệ thống công cụ mở rộng (Extensible Tool System):**
    - Kiến trúc cho phép dễ dàng thêm các "công cụ" mới để tương tác với nhiều dịch vụ AWS khác nhau (EC2, VPC, S3, RDS, IAM...).
    - Mỗi công cụ là một hành động cụ thể (ví dụ: `create-ec2-instance`, `list-vpcs`) mà Agent có thể gọi.

4.  **Quản lý trạng thái (State Management):**
    - Hệ thống tự động quét và duy trì một bản ghi về trạng thái của hạ tầng.
    - Điều này giúp AI đưa ra các quyết định chính xác, có nhận thức về ngữ cảnh và tránh xung đột (ví dụ: không cố tạo một tài nguyên đã tồn tại).

5.  **Kiến trúc API-First (API-First Architecture):**
    - Backend được xây dựng trên FastAPI, sẵn sàng để tích hợp với các giao diện người dùng tùy chỉnh hoặc các hệ thống tự động hóa khác thông qua REST API và WebSockets.

6.  **Quản lý người dùng và Bảo mật (User and Security Management):**
    - Tận dụng Supabase để cung cấp hệ thống xác thực người dùng an toàn (đăng nhập, đăng ký).
    - Mỗi hành động quản lý hạ tầng được gắn với một người dùng cụ thể, cho phép việc ghi log và truy vết (auditing).
    - (Mở rộng trong tương lai) Có thể xây dựng hệ thống phân quyền (RBAC - Role-Based Access Control) để giới hạn quyền hạn của từng người dùng trên hạ tầng AWS.

## Phần 3: Triển khai và Kiến trúc

### 3.1. Tổng quan triển khai (Implementation Overview)

Dự án được xây dựng với các thành phần và công nghệ chính sau:

- **Backend:**
    - **Ngôn ngữ:** Python 3.9+
    - **Framework:** FastAPI
    - **Quản lý môi trường:** Poetry
    - **Tương tác với AI:** LangChain (để làm việc với các LLM)
    - **Tương tác với AWS:** Boto3 (AWS SDK for Python)
- **Frontend:**
    - **Framework:** Next.js (React)
    - **Styling:** Tailwind CSS, Shadcn/UI
- **AI Providers:** Hỗ trợ nhiều nhà cung cấp LLM như Google (Gemini), OpenAI, Anthropic (Claude), và AWS Bedrock.
- **Xác thực & Lưu trữ dữ liệu:** Supabase được sử dụng để quản lý định danh người dùng (authentication) và lưu trữ siêu dữ liệu (metadata) về hạ tầng.

### 3.2. Kiến trúc triển khai (Deployment Architecture)

Sơ đồ dưới đây mô tả kiến trúc triển khai của hệ thống trên nền tảng AWS, tích hợp với các dịch vụ bên thứ ba.

```
+------------------+      HTTPS      +-----------------------+
|                  | <-------------> |                       |
|       User       |                 |   AWS Amplify (Web    |

+------------------+                 +-----------+-----------+
                                                 |
                                     HTTP/WebSocket|
                                                 v
                  +------------------------------+------------------------------+
                  |                  AWS Cloud (VPC)                            |
                  |                                                             |
                  |  +-------------------------------------------------------+  |
                  |  | AWS Fargate (Serverless Container)                    |  |
                  |  |                                                       |  |
                  |  | +---------------------------------------------------+ |  |
                  |  | |                                                   | |  |
                  |  | |         FastAPI Backend Container                 | |  |
                  |  | |                                                   | |  |
                  |  | +---------------------------------------------------+ |  |
                  |  |                                                       |  |
                  |  +------------------+------------------------------------+  |
                  |                     |                                       |
                  +---------------------+---------------------------------------+

|                                       |                                       |
v                                       v                                       v
+------------------+---------+  +-------+----------+-------------+  +-----------+--------------+
|                              |  |                                |  |                          |
|   Large Language Models      |  |   Amazon Web Services (AWS)    |  |  Supabase (Auth & DB)    |
| (Gemini, OpenAI, etc.)       |  |   (User's Target Environment)  |  |                          |
|                              |  |                                |  |                          |
+------------------------------+  +--------------------------------+  +--------------------------+
```

**Giải thích:**
1.  **User & Frontend:** Người dùng truy cập giao diện web (Next.js), được triển khai trên **AWS Amplify**, một dịch vụ hosting tối ưu cho các ứng dụng web hiện đại, cung cấp CI/CD và CDN toàn cầu.
2.  **Backend:** Ứng dụng FastAPI được đóng gói thành container và triển khai trên **AWS Fargate**. Đây là một giải pháp serverless cho container, giúp tự động scale và loại bỏ gánh nặng quản lý máy chủ.
3.  **Tương tác:**
    - Frontend (Amplify) giao tiếp với Backend (Fargate) qua HTTPS và WebSocket.
    - Backend gọi đến các dịch vụ **LLM** để xử lý ngôn ngữ tự nhiên.
    - Backend sử dụng **Supabase** để xác thực người dùng và lưu trữ siêu dữ liệu về hạ tầng.
    - Backend sử dụng **AWS SDK (Boto3)** để thực thi các hành động trên môi trường AWS của người dùng.

## Phần 4: Chi tiết triển khai (Implementation Details)

Để minh họa chi tiết, chúng ta sẽ phân tích chức năng: **"Tạo một EC2 instance"**.

- **Chức năng của thành phần:**
    - **Frontend:** Cung cấp ô chat để người dùng nhập lệnh. Gửi yêu cầu (kèm token xác thực) đến backend và hiển thị kết quả/tiến trình trả về qua WebSocket.
    - **Backend (API):** Nhận yêu cầu, sử dụng token để xác thực người dùng qua Supabase. Sau khi xác thực, chuyển yêu cầu cho AI Agent.
    - **AI Agent:** Tạo prompt, gọi LLM để nhận kế hoạch JSON.
    - **Plan Executor:** Thực thi kế hoạch, ví dụ gồm 2 bước: (1) Lấy AMI, (2) Tạo EC2.
    - **Tool & Adapter:** `CreateEC2InstanceTool` và `EC2Adapter` được gọi để thực hiện lệnh `run_instances` của Boto3.
    - **Supabase:** Lưu trữ thông tin người dùng, và sau khi tài nguyên được tạo, backend sẽ ghi siêu dữ liệu (ví dụ: ID của instance, người tạo, thời gian tạo) vào database của Supabase.

- **Các công việc đã thực hiện để hoàn thành chức năng:**
    1.  **Code Backend:**
        - Viết middleware hoặc dependency trong FastAPI để xác thực token của người dùng với Supabase trước khi xử lý bất kỳ yêu cầu nào.
        - Viết code trong `agent_router.py` để tạo endpoint `/api/v1/agent/process`.
        - Viết logic trong `agent.py` (`process_request`) để xây dựng prompt và gọi LLM.
        - Viết logic trong `plan_executor.py` để đọc kế hoạch, giải quyết các tham số phụ thuộc và gọi tool.
        - Sau khi tool thực thi thành công, viết logic để ghi log hoặc metadata vào bảng dữ liệu trên Supabase.
        - Viết `CreateEC2InstanceTool` và `EC2Adapter` để bao bọc lệnh gọi Boto3.
    2.  **Code Frontend:**
        - Tích hợp Supabase Auth vào ứng dụng Next.js để quản lý đăng nhập, đăng ký.
        - Lưu token xác thực của người dùng và đính kèm nó vào header của mỗi yêu cầu gửi đến backend.
        - Xây dựng giao diện chat và logic WebSocket.
    3.  **Cài đặt & Thiết lập:**
        - Tạo một project trên Supabase, lấy URL và service key.
        - Thiết kế schema cho các bảng dữ liệu trên Supabase (ví dụ: bảng `users`, bảng `resources`).
        - Cấu hình môi trường Python bằng `poetry`.
        - Thiết lập file `.env` để lưu các API key (AWS, Google/OpenAI, Supabase).
        - Đảm bảo môi trường chạy có quyền truy cập AWS.

- **Giá trị của công việc:**
    - Công việc không chỉ dừng lại ở việc "viết code".
    - **Thiết kế giải pháp:** Thiết kế luồng đi từ ngôn ngữ tự nhiên -> kế hoạch JSON -> thực thi là một phần cốt lõi.
    - **Prompt Engineering:** Xây dựng các prompt hiệu quả để LLM hiểu và trả về kết quả mong muốn là một công việc đòi hỏi nhiều thử nghiệm và tinh chỉnh.
    - **Kiến trúc hệ thống:** Xây dựng hệ thống theo các pattern (Factory, Adapter) để dễ dàng mở rộng và bảo trì trong tương lai.

## Phần 5: Quá trình thực thi dự án (Project Execution)

### 5.1. Lịch sử phát triển dự án

Dự án được bắt đầu với ý tưởng đơn giản hóa việc quản lý AWS. Quá trình phát triển trải qua nhiều giai đoạn với các quyết định kỹ thuật quan trọng:
- **Tuần 1-2: Khởi đầu và định hình kiến trúc.**
  - Ban đầu, nhóm đã xem xét phương pháp tiếp cận dựa trên script đơn giản. Tuy nhiên, để đạt được sự linh hoạt và thông minh thực sự, nhóm đã quyết định chuyển hướng sang một kiến trúc AI Agent phức tạp hơn, với khả năng tự lập kế hoạch.
  - Lựa chọn công nghệ chính: FastAPI cho backend vì hiệu năng cao và hệ thống dependency injection mạnh mẽ; Next.js cho frontend vì khả năng server-side rendering và hệ sinh thái phong phú.

- **Tuần 3-4: Xây dựng "bộ não" của Agent.**
  - Tích hợp LangChain và thử nghiệm với các LLM khác nhau.
  - **Thách thức lớn nhất:** Đảm bảo LLM luôn trả về một kế hoạch JSON hợp lệ và có thể thực thi được. Nhóm đã phải đối mặt với các vấn đề như LLM "ảo giác" ra các tham số không tồn tại hoặc trả về JSON sai định dạng.
  - **Giải pháp:** Áp dụng kỹ thuật "few-shot prompting" (cung cấp ví dụ trong prompt), xây dựng logic để xác thực và làm sạch (clean) response từ LLM, và tinh chỉnh prompt liên tục để hướng dẫn LLM một cách chặt chẽ nhất.

- **Tuần 5-6: Phát triển hệ thống Tool và Adapter.**
  - **Quyết định kiến trúc:** Nhóm đã quyết định tách biệt rõ ràng giữa `Adapters` (lớp mỏng, chỉ chịu trách nhiệm gọi Boto3) và `Tools` (lớp chứa logic nghiệp vụ và được Agent sử dụng). Quyết định này giúp hệ thống trở nên cực kỳ modular, dễ dàng thêm tool mới mà không ảnh hưởng đến các phần khác.
  - Bắt đầu với các dịch vụ cốt lõi: EC2 và VPC.

- **Tuần 7-8: Xây dựng cơ chế thực thi và giao diện người dùng.**
  - **Thách thức:** Làm thế nào để thực thi một kế hoạch có các bước phụ thuộc vào nhau (ví dụ: dùng ID của VPC vừa tạo để tạo subnet).
  - **Giải pháp:** Xây dựng cơ chế "context passing" trong `PlanExecutor`, cho phép các bước sau tham chiếu đến output của các bước trước thông qua cú pháp `{{step-id.output-key}}`.
  - Tích hợp WebSocket để cập nhật giao diện người dùng theo thời gian thực, mang lại trải nghiệm mượt mà và trực quan.

- **Tuần 9-10: Mở rộng, Tối ưu và Hoàn thiện.**
  - Mở rộng hệ thống với các tool cho RDS, S3, và Security Group.
  - Tích hợp Supabase để quản lý xác thực, chuẩn bị cho việc lưu trữ và truy vết.
  - Tối ưu hóa các câu lệnh prompt và rà soát lại toàn bộ luồng hoạt động để sửa lỗi và cải thiện độ tin cậy. Chuẩn bị tài liệu cho báo cáo cuối kỳ.
